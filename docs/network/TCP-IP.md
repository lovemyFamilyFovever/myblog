严格来说三次握手是 **TCP** 的，不是 HTTP 的。HTTP 是建立在 TCP 之上的应用层协议。

## 先理解为什么需要握手

```
你要和朋友打电话
    ↓
你拨号 → 对方接听 → 你确认对方在听 → 开始聊天
    ↓
如果直接说话，对方可能没准备好，听不到开头
```

TCP 三次握手就是这个道理 — 在正式传输数据之前，双方确认彼此都准备好了。

## 三次握手的过程

```
客户端                              服务器
  │                                   │
  │                                   │
  │──── 第一次: SYN ────────────────→│  "你在吗？我想连接"
  │     seq=100                       │
  │                                   │
  │←─── 第二次: SYN + ACK ──────────│  "我在，我也准备好了"
  │     seq=300, ack=101              │
  │                                   │
  │──── 第三次: ACK ────────────────→│  "好的，那我们开始"
  │     ack=301                       │
  │                                   │
  │         连接建立，开始传数据         │
```

### 第一次握手

```
客户端 → 服务器
    ↓
SYN 标志位 = 1
seq = 100（客户端随机生成的初始序列号）
    ↓
含义：我想和你建立连接，我的起始编号是 100
    ↓
客户端状态：SYN_SENT（已发送，等待回复）
```

### 第二次握手

```
服务器 → 客户端
    ↓
SYN 标志位 = 1
ACK 标志位 = 1
seq = 300（服务器自己的初始序列号）
ack = 101（确认收到了客户端的 100，期望下一个收到 101）
    ↓
含义：收到你的请求了，我也想和你连接，我的起始编号是 300
    ↓
服务器状态：SYN_RECEIVED（已收到，等待确认）
```

### 第三次握手

```
客户端 → 服务器
    ↓
ACK 标志位 = 1
ack = 301（确认收到了服务器的 300，期望下一个收到 301）
    ↓
含义：收到你的确认了，我们可以开始传数据了
    ↓
双方状态：ESTABLISHED（连接已建立）
```

## 为什么是三次，不是两次

```
两次握手的问题

客户端 → 服务器: 我想连接
服务器 → 客户端: 好的，连接建立
    ↓
如果第一次的请求是网络中滞留的旧请求
    ↓
服务器以为是新连接，白白等着
    ↓
资源浪费（这就是 SYN 洪泛攻击的原理）
```

```
三次握手解决了这个问题

客户端 → 服务器: 我想连接
服务器 → 客户端: 收到了，你确认一下
客户端 → 服务器: 我确认，开始吧
    ↓
如果客户端根本没发起过连接
    ↓
不会发第三次握手
    ↓
服务器知道这是无效请求，释放资源
```

## 用生活场景类比

```
打电话

第一次: 你打电话 → "喂，小王在吗？"
第二次: 对方接 → "在的，你是？"  
第三次: 你确认 → "我是小李，咱们说个事"
    ↓
正式开始通话
```

```
如果只有两次

你: "喂，小王在吗？"
对方: "在的"
    ↓
但你可能是打错了电话
对方一直等着你说话
    ↓
资源浪费
```

## seq 和 ack 的作用

```
seq（序列号）: 标记我发的数据从哪开始编号
ack（确认号）: 告诉对方我期望收到的下一个编号

为什么需要编号？

发送方: [包1][包2][包3][包4][包5]
                    ↓
网络传输中包2丢了
                    ↓
接收方: [包1][包3][包4][包5]
                    ↓
发现编号不连续，请求重传包2
                    ↓
发送方重传: [包2]
                    ↓
接收方: [包1][包2][包3][包4][包5] ✓
```

## 完整的连接和断开

```
建立连接: 三次握手 (SYN → SYN+ACK → ACK)
    ↓
传输数据: 双向发送
    ↓
断开连接: 四次挥手 (FIN → ACK → FIN → ACK)
```

```
四次挥手

客户端 → 服务器: FIN  "我说完了"
服务器 → 客户端: ACK  "知道了，但我可能还有数据要发"
服务器 → 客户端: FIN  "我也说完了"
客户端 → 服务器: ACK  "好的，断开吧"
    ↓
为什么断开要四次？
    ↓
因为连接是双向的
建立时双方同时确认，可以合并
断开时可能一方还有数据没发完，不能合并
```

## 总结

| 握手 | 方向 | 内容 | 目的 |
|---|---|---|---|
| 第一次 | 客户端 → 服务器 | SYN | 发起连接请求 |
| 第二次 | 服务器 → 客户端 | SYN + ACK | 确认请求并回应 |
| 第三次 | 客户端 → 服务器 | ACK | 确认收到回应 |

**核心目的：双方确认彼此的发送和接收能力都正常，然后才开始传数据。**
